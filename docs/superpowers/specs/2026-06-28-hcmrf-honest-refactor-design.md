## 1. 问题背景

### 1.1 识别的问题

经过全面审阅，发现以下问题：

1. **消融实验漏洞**：90天预测的Patch消融无意义（因为90天本就用patch=1，消融对比的是模型自己）
2. **通道门控无效**：去掉Gate后MSE反而略降（374k→372k），说明该组件学不到有用特征
3. **硬编码分支限制创新**：课程要求"分别训练"导致真正的horizon conditioning数学上退化（每个模型实例只看到一个horizon，if-else是常数而非学习）
4. **超参数缺乏依据**：压缩因子=3、kernel=7等魔数没有消融支撑

### 1.2 根本矛盾

课程硬性要求"90d和365d模型分别训练"，这意味着：
- 每个模型实例只看到一个horizon
- 任何"horizon-aware"设计实际上都是硬编码常数
- 真正的learned horizon conditioning（如FiLM）需要模型同时见过多个horizon

**结论**：在课程约束下，无法实现真正的learned horizon conditioning。

## 2. 设计方案：诚实重构

### 2.1 核心叙事修正

| | 原来（有数学缺陷） | 修正后（诚实） |
|---|---|---|
| **名称** | Horizon-**Conditioned** Multi-Resolution Forecasting | Horizon-**Specialized** Multi-Resolution Forecasting |
| **核心论点** | "模型学会感知预测距离并动态调整" | "根据多分辨率原理，为不同horizon设计专门化架构，每个专门化都由消融验证" |
| **Horizon如何起作用** | 声称是learned（实际是硬编码常数） | 明确是architectural design choice（设计决策，非学习） |

**关键诚实点**：报告的"未来工作"里明确写出"受课程分别训练约束限制，真正的learned horizon conditioning不可行，理想方案是FiLM+多任务单模型，留作未来工作"——把局限变成洞察，反而是加分项。

### 2.2 组件决策

| 组件 | 决策 | 理由 |
|---|---|---|
| **DRD（粗→精解码）** | ✅ 保留，作为**核心卖点** | 消融证明去掉后+46% MSE，是唯一大效果组件 |
| **多尺度池化**（365d池化到30步） | ✅ 保留 | +7%效果，符合"长horizon需粗粒度"原理 |
| **Patching**（365d patch=3） | ✅ 保留 | +5%效果 |
| **通道门控Gate** | ❌ **删除** | 实验证明无效（去掉反而略降），保留是自欺 |
| **共享编码器** | ✅ 保留 | 减参数 |

### 2.3 消融实验重新设计

**核心问题修复**：90d路径本就用patch=1且不池化，所以Patch/多尺度消融对90d无意义。修正方案——**消融只针对365d报告，90d作为"细分辨率基线架构"明确说明**：

| 消融变体 | 365d改动 | 验证的问题 |
|---|---|---|
| **完整 MRF** | — | 基线 |
| **-w/o DRD** | 直接Dense(365)替代粗→精 | 粗→精解码是否关键（预期大幅下降） |
| **-w/o MultiScale** | 不池化，保持90步 | 多分辨率是否有用 |
| **-w/o Patch** | patch=1替代patch=3 | patching是否有用 |
| **-w/o Shared** | 独立编码器（参数翻倍） | 共享编码器是否有效 |

**注意**：所有消融仅对365d有意义。90d路径不池化、patch=1，因此MultiScale/Patch消融对90d退化为恒等操作。报告中消融表格只列365d结果，90d单独说明。

**新增超参数消融**（为魔数提供依据）：
- 池化压缩因子：{2, 3, 4}
- DRD精修kernel：{3, 5, 7}
- 粗预测周数：{26(半月), 52(周)}
